"# Chapter 9: Bemi v2.0: Scaled Dominance Architecture\n\n## 9.1 Adaptive Thread Scheduling (84 to 48)\n\n### 9.1.1 The Pitfalls of Thread Oversubscription\nIn the early design phases of the Bemi BIOS (exemplified by version 1.2 and 1.3), the architecture prioritized high hardware thread density as the primary mechanism for bypassing memory latency. By presenting up to 144 virtual threads (in v1.2) or 84 virtual threads (in v1.3) to the guest Operating System, the scheduler aimed to hide cache-miss latency by rapidly rotating executing threads. The assumption was that the sheer volume of threads would guarantee that at least one thread would always be ready to execute on the physical Arithmetic Logic Units (ALUs).\n\nHowever, physical implementation under realistic silicon constraints revealed a catastrophic trade-off: **L1 Cache Thinning**. \n\nModern processor caches are fabricated with high-density SRAM cells. Because physical die area is strictly limited by semiconductor economics, the total L1 cache budget per core is capped (typically 32 KB for instructions and 32 KB for data). When this budget is sliced across 84 virtual threads, each thread is allocated a meager $4.57 \\text{ KB}$ of L1 cache. When sliced across 144 threads, this drops to a microscopic $2.67 \\text{ KB}$. \n\nThis cache thinning triggers **thrashing**. As threads are swapped in and out of the execution units, their working sets constantly overwrite each other in the tiny L1 cache. The cache miss rate spikes, converting the intended latency-hiding mechanism into a latency-generation engine.\n\n### 9.1.2 Rebalancing the Thread-to-Cache Ratio\nTo resolve this thrashing, Bemi v2.0 introduces the **Adaptive Thread count** model.
